## Context and Orientation

当前仓库是多服务结构，Admin 前端位于 `apps/admin-web`，后端主要由 `services/gateway-bff`、`services/identity`、`services/commerce`、`services/payment` 组成。对齐工作需要同时看三层：

第一层是页面是否真实调用 API。关键入口是 `apps/admin-web/src/react/pages/admin/*.tsx` 与 `apps/admin-web/src/lib/api.js`。

第二层是网关是否有路由入口。关键文件是 `services/gateway-bff/internal/http/server.go` 与 `services/gateway-bff/internal/http/proxy.go`。

第三层是契约与实现是否存在。关键文件是 `contracts/openapi/*.yaml` 与各服务 `internal/http`、`internal/modules`。

本清单中的“缺少”，特指 Admin 页面需求在后端侧没有可直接对接的稳定契约或实现。

## 后端缺口清单（相较 Admin 前端）

### P0：应优先补齐（阻塞业务闭环）

#### 1) 管理员列表聚合接口缺失

用户运营页面已有“管理员用户”区块，但明确写明待接入。当前后端没有面向 Admin 的管理员列表聚合接口。

建议新增能力：`GET /admin/users?role=ADMIN&page=&pageSize=&q=`，返回统一分页结构并附角色、状态、最后登录时间等运营字段。

#### 2) 客户直接提权为业务员流程缺失

页面当前调用 `patchStaffRoles(customer.id, ...)`，当客户尚无 staff 记录时会走 404 分支并提示“后端暂未开放此能力”。

建议新增能力：`POST /admin/customers/{customerId}/promote-to-sales`。接口内部应完成“客户身份到员工身份”的幂等转换，并支持重复调用安全返回。

#### 3) 需求单详情（Requirement Profile）接口缺失

询价页面现在用 `PriceInquiry` + `InquiryMessage` 能拿到基础会话，但右侧“需求单画像”字段（公司名、联系人、期望数量、目标单价等）仍靠 mock 兜底。

建议新增能力：`GET /admin/inquiries/{inquiryId}/requirement-profile`，返回页面所需完整结构，避免前端拼装或硬编码默认值。

#### 4) 支付运营与审计接口缺失

Admin 支付页是“交易 + 审计 + Webhook”三块视图；现有支付服务仅有创建支付和微信回调，无法支持运营查询。

建议新增能力：
`GET /admin/payments/transactions`、`GET /admin/payments/transactions/{id}`、`GET /admin/payments/audit-logs`、`GET /admin/payments/webhooks`、`POST /admin/payments/webhooks/{id}/replay`。

### P1：领域后端尚未建立（页面已存在）

#### 5) 供应商域缺失

`SuppliersPage` 已有完整管理界面，但后端无 supplier 领域契约与实现。

建议最小可用契约：`/admin/suppliers`（列表/创建）、`/admin/suppliers/{id}`（详情/更新）、`/admin/suppliers/{id}/scorecards`（履约评分）、`/admin/suppliers/{id}/contacts`。

#### 6) 报价工作流域缺失

`QuoteWorkflowPage` 展示了版本流、审批流与对话区，但后端无“报价版本与状态机”契约。

建议最小可用契约：`/admin/quotes`、`/admin/quotes/{quoteId}`、`/admin/quotes/{quoteId}/versions`、`/admin/quotes/{quoteId}/submit`、`/admin/quotes/{quoteId}/approve`、`/admin/quotes/{quoteId}/reject`。

#### 7) 客服工作台域缺失

`SupportWorkspacePage` 包含会话列表、消息流、客户画像、订单上下文，但后端缺少面向工作台的会话聚合接口。

建议最小可用契约：`/admin/support/conversations`、`/admin/support/conversations/{id}`、`/admin/support/conversations/{id}/messages`、`/admin/support/conversations/{id}/assign`、`/admin/support/conversations/{id}/close`。

#### 8) 系统设置持久化接口不足

`SettingsPage` 为配置中心 UI，但当前后端仅有 feature flags 相关接口，无法覆盖账号策略、通知策略、白名单、审计配置等设置项。

建议最小可用契约：`GET/PUT /admin/settings/general`、`GET/PUT /admin/settings/security`、`GET/PUT /admin/settings/notifications`、`GET /admin/settings/history`。

### P2：统计与聚合接口增强

#### 9) 仪表盘专用分析接口不足

仪表盘已接 `/bff/admin/summary`，但页面仍提示“RBAC 分析接口未接入”“导入任务专用统计未接入”。说明当前 summary 维度不足。

建议新增能力：`GET /bff/admin/analytics/rbac` 与 `GET /bff/admin/analytics/import-jobs`，避免前端拼装统计。

#### 10) 物流与询价的“实时流”能力不足（按产品目标决定）

仓库中旧版 dev 页面仍存在“实时物流详情流/实时询价详情流未接入”提示。若产品要求是真实时序刷新（SSE 或轮询聚合），后端需提供统一流式或增量接口；若接受普通查询，则应明确复用现有 `/orders/{orderId}/tracking` 与 `/inquiries/price/{id}/messages`。

## 非后端缺失（避免误判）

以下能力后端已存在契约，但 React 管理页仍以静态数据展示，属于前端接线任务，不计入后端缺口：

订单基础数据（`/orders`）、物流跟踪基础数据（`/orders/{orderId}/tracking`）、商品与类目（`/catalog/*`）、RBAC 基础能力（`/rbac/*`）、客户转移与标签（`/admin/customers/transfer`、`/admin/customer-tags` 等）。

## Plan of Work

第一阶段先做“契约冻结”。在 `contracts/openapi` 增加或扩展 admin 领域路径，明确请求参数、返回结构、权限要求和幂等语义。这里不要直接写实现，先确保前后端对字段定义一致。

第二阶段做“网关打通”。在 `services/gateway-bff/internal/http/server.go` 挂载新增 admin 路由，并在 `proxy` 层完成转发与错误语义统一（4xx/5xx、request-id、审计埋点）。

第四阶段做“联调回归”。逐页替换前端 mock 分支，保证页面在 dev 模式和真实后端模式都可运行。每完成一个页面，就补一组最小回归用例，避免后续回退。

## Validation and Acceptance

本清单完成实现后，验收标准是“行为可见”而不是“代码存在”：

管理员打开用户运营页时，管理员列表区域应展示真实分页数据，不再显示“待接入”。

管理员在客户列表点击“设为业务员”时，真实模式应返回成功并刷新员工列表，不再依赖 mock 或 404 分支提示。

管理员打开询价页时，需求单画像应来自后端接口，不再出现默认兜底值。

管理员打开支付页时，可按交易号查询、查看审计日志与 webhook 事件，并可执行重放动作。

管理员打开供应商、报价工作流、客服工作台、系统设置页时，页面数据应全部来自后端接口，刷新后可保持状态一致。

## Idempotence and Recovery

契约变更采用增量方式新增路径，避免破坏已有调用。若某个新接口实现延期，可先返回明确的 `501 not_implemented` + 结构化错误码，而不是静默失败。

“客户提权为业务员”接口必须保证幂等，重复提交同一客户不会产生重复 staff 记录。若流程中断，可通过同一请求安全重试。

联调阶段建议逐页开关发布（feature flag），出现线上问题时可以按页面回退到旧逻辑，不影响其他已完成页面。

## Interfaces and Dependencies

建议由 `contracts/openapi/openapi.yaml` 统一聚合新增路径，再分别下沉到 `identity.yaml`、`commerce.yaml`、`payment.yaml`。所有新增 admin 接口统一遵守以下规则：

鉴权沿用 Bearer JWT 与现有 RBAC 权限模型。写操作必须携带 request-id 并进入审计日志。

分页结构统一为 `items/page/pageSize/total`，避免各页面适配多套响应格式。

错误语义统一为结构化错误对象（`code`、`message`、`requestId`），并与网关转发行为一致。

涉及“状态流转”的接口（报价审批、会话关闭、客户提权）必须明确定义状态机与非法状态返回码，避免前端靠文案猜测。
