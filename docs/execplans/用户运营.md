# 用户运营页优化清单

## 目标
沉淀 `user-operations` 页面的中优先级与结构性优化项，作为后续迭代执行基线。

## 中优先级优化

### 1) 客户标签筛选控件可用性
- 问题：当前使用原生 `multiple select`，运营同学在大量标签场景下难以搜索和确认已选项。
- 影响：筛选效率低，误操作概率高。
- 建议方案：改为可搜索多选（支持标签 chips + 一键清空）。
- 验收口径：支持键盘检索、可视化已选标签、清空筛选不刷新整页。

### 2) 全局提示语义分域
- 问题：`errorMessage/successMessage` 在页面级共享，跨 tab 操作时会出现历史提示残留。
- 影响：运营判断当前操作状态成本高。
- 建议方案：按 tab 或 action 维度维护提示状态，或统一 toast 队列。
- 验收口径：切换 tab 不出现无关提示；提示内容明确包含动作主体。

### 3) Loading 反馈颗粒度
- 问题：表格区域主要依赖按钮文案反馈，加载中与空状态视觉接近。
- 影响：用户难区分“正在请求”与“没有数据”。
- 建议方案：引入骨架屏/占位行，并区分首次加载、刷新加载、空状态。
- 验收口径：首屏与翻页均有稳定 loading 占位；空状态仅在请求结束后展示。

## 结构性优化

### 1) 页面拆分与职责解耦
- 问题：`UserOperationsPage.tsx` 同时包含数据请求、状态机、数据归一化与 UI 渲染，文件过大。
- 影响：变更风险高，回归成本高。
- 建议方案：
  - 抽离 `useUserOperationsData`（请求、竞态、分页、局部更新）
  - 抽离 `CustomerPanel / StaffPanel / AdminPanel`
  - 抽离 `normalizers.ts`
- 验收口径：页面主组件仅负责组合，业务行为由 hooks + 面板组件承载。

### 2) 统一状态与动作模型
- 问题：客户/员工/管理员三块的分页、过滤、pending、提示语管理风格不统一。
- 影响：后续新增动作（如批量操作）时重复代码多。
- 建议方案：统一为“列表查询状态 + action pending map + 请求版本控制”模式。
- 验收口径：三个 tab 的状态更新路径一致，可复用工具函数。

### 3) 测试资产分层
- 问题：当前页面行为主要依赖 e2e 覆盖，缺少面向 hooks/局部更新策略的快速测试。
- 影响：小改动需依赖重型回归。
- 建议方案：补充 hooks/组件层测试（竞态丢弃、局部更新、分页参数）。
- 验收口径：关键数据行为可在非 e2e 层快速验证。

## 后续建议
- 下一轮按“体验（中优先级） -> 架构（结构性）”推进，避免与核心业务需求并行造成回归面扩大。
